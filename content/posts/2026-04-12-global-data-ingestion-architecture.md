# 글로벌 데이터 수집 파이프라인 설계

### 1. 해외 데이터 수집 설계

>  국내 혹은 아시아 권역 수집이 아니라 태평양 너머의 데이터 수집의 경우 거리에 따른 네트워크 이슈가 가장 큰 도전과제이며 이에 따른 문제를 경험해보지 않았기 때문에 이 문제를 해소하기 위한 검토 사항을 가장 우선적으로 설계에 반영해야만 한다

#### 1-1. 검토 사항

* 대만 해저 케이블 단절과 같은 상황이 발생하여 일정 시간 혹은 기간 동안 네트워크 장애 발생 시에도 **데이터 유실이 없어**야 한다
* 기존 YARN 기반의 **안정적인 작업 스케줄링, 상태 모니터링** 및 **자원 (코어, 메모리 + 네트워크) 관리가 가능**해야 한다
* 네트워크 상황이 안정적이지 않아서 비정기적으로 단절 혹은 지연이 발생하더라도 **이미 전송한 데이터를 최대한 다시 전송하지 않아**야 한다
* 원격지 내의 네트워크 리소스 사용 또한 대역폭이 충분하지 않아서  **특정 프로세스 혹은 작업이 전체 네트워크 대역을 과점유하는 상황이 없어**야 한다
* 네트워크 리소스는 공유 자원이므로 짧게 굵게 보다는 **길고 가늘게 사용하는 전략이 유효하며, 이를 전체 사용 수준에서 조절할 수 있어**야 한다
* 리소스 혹은 여러 이유로 인한 **작업 실패에 대한 고가용성**, **장비 추가를 통한 수평적 성능 향상** 및 내결함성을 보장하는 데이터 파이프라인

#### 1-2. 설계 방향

* 네트워크 환경이 안정적인 구간과 그렇지 않은 구간의 **네트워크 프로토콜을 구분**하여 설계
  * 북미 권역 내에 수집은 데이터베이스 테이블 접근 API 인 JDBC 활용하여 개별 데이터베이스 별 프로토콜 도구 `apache-sqoop` 
  * 대륙간 수집은 파일 및 디렉터리 동기화 프로토콜 도구 `rsync` 활용하여 송수신 과정에서 <u>네트워크 장애가 발생하더라도 재전송 없음</u>
* **네트워크 지연 및 회선 장애 시에도 독립적인 스케줄링**을 통해 수집 가능하도록 설계
  * 테이블 스냅샷 특성 상 스냅샷 시점에 원본에서 데이터 수집이 이루어져야 하므로 <u>수집 스케줄러는 원격지에서 기동 및 동작</u>
  * 운영 자동화, 모니터링 및 장애 알람을 위해서 <u>국내의 빅브라더 내에서 모든 메타데이터 및 상태 관리가 가능하도록 설계</u>
* **운영 자동화 및 장애 인지는 국내 빅브라더 내에서 수행**될 수 있도록 설계
  * 원격지의 데이터베이스 및 장비 목록 확인을 위한 ACL 오픈 및 접근이 가능해야 한다
  * 테이블 및 파일 수집을 위한 <u>메타데이터 등록 및 스케줄링은 국내 빅브라더에서 수행</u>되어야 한다
  * 원격지에서 실행 중인 <u>작업을 종료하고 다시 실행하려고 하는 경우에도 동작 및 모니터링</u> 하도록 설계
* **네트워크 자원을 효율적이고 안정적으로 사용**할 수 있는 데이터 파이프라인 설계가 되어야 한다
  * 데이터 수집 시에 동시에 실행되거나 과도한 데이터 수집 요청이 발생하지 않도록 설계
  * 한 번에 모든 데이터를 가져오거나 <u>네트워크 버스팅이 발생하지 않도록 슬로우 스타트</u> 가능한 스케줄러 개발
  * 개별 테이블 수준이 아니라, 전체 네트워크 대역을 고려한 <u>수집 대상 테이블 전체에 대한  스로틀링이</u> 가능하도록 설계
    * 기존 YARN 과 같이 전체 리소스 기준으로 스케줄링 하는 방식의 가장 큰 단점은 전체 네트워크 사용 제어가 어려움
* **고가용성, 확장성 및 내결함성을 보장하는 데이터 전송 파이프라인**을 설계해야 한다
  * 원격지의 수집 클러스터의 경우에도 네트워크 뿐만 아니라 <u>장비의 물리적인 장애에도 고가용성</u> 확보
  * 접속자의 증가 및 데이터 자체가 커지는 경우에도 <u>물리적인 장비의 추가 만으로 선형적인 수집 성능 증가가 가능한 확장성</u> 확보
  * 개별 <u>장비 혹은 컨테이너 수준의 장애 시에도 정합성이 보장</u>될 수 있는 내결함성 확보

### 2. 수집 시스템 설계

#### 2-1. 데이터 제어 API

* 각종 메타데이터는 빅브라더 한국 서버에서 관리하고, 이를 복제한 데이터를 전송하여 상태 관리만 원격지에서 하도록 설계한다

  * 테이블 수집은 데이터베이스 정보와 테이블 정보를 전달한다

  * 게임 로그는 게임 로그 서버 정보와 테이블 정보를 전달한다

* 빅브라더 작업 수행 시에는 기존 메타데이터에서 별도의 `flag` 값을 통해 구별하여 `executeProcess` 함수 내에서 별도로 구현한다

  * 테이블 수집 등록 시에 기본 설정을 국내 수집으로 하고 별도의 체크를 통해서 글로벌 데이터 `NCA` 수집 플래그를 추가한다
  * 수집 API 호출 시에 수집에 필요한 모든 메타를 담은 정보를 원격 API 통해 전달하고 별도 몽고 디비에 저장
  * 몽고에 저장된 테이블 수집 메타를 통해서 테이블 수집을 표준 프로퍼티 파일에 저장하는 기능
  * 테이블 수집은 수시로 호출 될 수 있고 등록되더라도 원격지에서 버퍼링 하여 스케줄링 하고
  * 게임 로그 수집은 한 번 호출에 수집 대상 전체를 등록하되 원격지에서 슬로우 스타트 하여 스케줄링 해야 한다

#### 2-2. 잡 스케줄러

* <u>도커 스웜을 통한 스케줄링 및 컨테이너 제어</u> 기능
  * 리소스 (코어, 메모리 및 네트워크 등)을 제어하는 컨테이너 실행 기능 - 코어 및 메모리 사용에 따른 `spread` 전략 적용이 가능함
    * 테이블 수집 작업을 크게 3등급으로 구분하여  `heavy`, `medium`, `light` 로 구분하여 사전에 메타데이터를 관리
    * 스플릿이 들어간 무거운 작업은 `2core/8GB` 큰 테이블은 `1core/4GB` 나머지 테이블은 `0.5core/2GB` 등으로 등급을 구분하여 처리
  * 도커 스웜의 기능을 살펴보고, 컨테이너 통한 애플리케이션 실행을 스케줄링 하는 기능 (부족 시에 대기하는 등)
  * 테이블 수집의 경우 몽고에 저장된 메타데이터 목록을 조회하여 인터벌 시간마다 인입된 전체 테이블 수집 스케줄링을 최적화 하는 방향으로 수집
  * 게임 로그 수집의 경우 빅비트에 내장된 크론 스케줄링 정보를 통해서 주기적인 스케줄링을 수행
* 실행 한 <u>컨테이너의 상태를 모니터링하고, 시작, 종료 상태 확인 등을 API 수준에서 제어</u> 할 수 있는 기능
  * 작업 실행에 대한 타임아웃, 실패 시에 재시도 그리고 장애 시에 강제 복구 등의 기능
  * 동시 실행 임계치 혹은 네트워크 대역 범위를 초과하여 수행될 수 없도록 대기하는 기능
* <u>국내 빅브라더 서버의 상태와 연동된 스케줄링</u> 기능
  * 빅브라더의 로컬 상태 및 원격지 상태를 변경하는 기능
* <u>파이썬 기반의 도커 서비스 수준의 데몬 스케줄러</u> 기능
  * Docker 공식 문서 기준으로 Docker Engine은 공식 SDK를 Go와 Python에 제공
  * 몽고 폴링 도커 서비스 기동 및 관리 등을 위한 빠른 시간 내에 적용 가능한 포터블한 엔진 개발

#### 2-3. 테이블 수집 컨테이너

* 아파치 스쿱 컨테이너 이미지 내부에서 입력 인자로 받은 프로퍼티 파일을 읽고 로컬 저장소에 저장하는 기능
* 테이블 수집의 3가지 유형에 대해 동일한 방식으로 수집하는 기능
  * 하나의 파티션 테이블은 하나의 컨테이너를 통해 수집
  * 하나의 파티션이면서 다수의 스플릿 테이블은 하나의 컨테이너 내에서 다수의 프로세스를 통해서 수집
  * 다수의 파티션 테이블은 다수의 컨테이너 호출을 통해서 수집 (스케줄러 수준에서 분할 수행)

#### 2-4. 게임로그 수집 컨테이너

* rsync 명령어 수행하는 컨테이너 통해서 원격지에 저장된 파일을 로컬에 동기화하는 기능
* 로컬에 동기화된 파일을 다시 원격지에 동기화 하는 기능

* 국내 빅비트의 슬로우 스타트 및 스로틀링을 좀 더 유연하게 데이터 기반으로 수행할 수 있는 기능
  * 테이블 멀티 파티션 수집과 유사하게 스케줄러 수준에서 개별 작업을 실행시켜줘야 할 수 있음
