### 해외 데이터 수집 아키텍처 설계

* 배경
  * 대륙을 넘어선 네트워크 환경에서 대용량 테이블 수집과 같은 성공 실패 단위의 트랜잭션이 아주 큰 테이블 수집 작업은 JDBC 와 같은 프로토콜은 적합하지 않다
  * 클라우드 서비스의 경우에도 직접 접근을 하기 보다는 리전간 동기화 같은 별도의 서비스를 통해 제공하는 것을 통해 고려해야 할 사항이 많아질 수 있다
* 질문
  * 국내 게이트웨이 수준에서 연동도 가능할 것 같은데 NGINX 수준에서 해외 VIP 정보를 직접 등록해서 사용하면 어떨까?
    * 간편하게 연동이 가능할 수도 있지만 국내 게이트웨이 정보와 해외가 섞여서 장애 시에 제어가 더 어려울 수 있겠다
    * **UI 수준에서 서비스 연동을 하고 특정 서비스만 독립적으로 운영**하는 것이 아키텍처도 간결해지고 운영에도 좋을 듯 싶다
  * 테이블 수집을 로컬 저장소에 저장하고 국내로 동기화 하는 방안을 검토하는 경우 저장소는 어떻게 구성하는 게 좋을까?
    * 로컬 스토리지의 경우 제약이 있을 수도 있고 장애 시에 복구도 쉽지 않은 구조이므로 좋지 않은 방향 같다
    * **저장 시에 클라우드 블록 스토리지를 활용**할 수 있다면 비용적인 부분은 발생할 수 있으나 안정적인 저장 및 수집이 가능할 것이다
    * 다만 블록 스토리지에 직접 쓰기 시에 다양한 이슈가 발생할 수 있으며 기존 HDFS 기반 구현체가 정상동작하지 않을 수 있다
    * 결국 로컬 스토리지에 저장하고 해당 데이터를 일정 기간 보관하고 이를 마운트하여 가져오는 방법도 후보로 둘 필요가 있다
    * 클라우드 저장소라면 현재 **구현되어 있는 클라우드 저장소 수집기를 통해 국내로 가져오는 것도 용이**할 것이다
  * 원격지 테이블 수집 프로세스는 도커 스웜, 쿠버네티스 혹은 매니지드 서비스를 통한 컨테이너 기반 서비스는 어떨까?
    * 도커 이미지를 빌드 및 배포하여 원격지에서 컨테이너 수준에서 수행 가능하도록 포터블 한 구성이 좋겠다
    * 메타데이터 정보를 직렬화 하여 API 에 전달하고 이를 통해서 테이블을 저장소에 저장하는 실행기를 구현하면 좋겠다
  * 기존 구현된 얀 기반의 환경을 구축하고 기존 애플리케이션을 포팅하는 것이 적절한가?
    * 얀 클러스터를 구축하고 모니터링 운영하는 것 만으로도 충분히 부담이 되며 고정 장비를 지속적으로 켜 두는 것도 부담될 수 있다
    * 클라우드 리소스 상 새벽에 필요한 데이터에 대해서만 0시 ~ 9시 사이에만 사용하면 되는 상황이면 더 좋을 수 있겠다
  * 단일 테이블, 서버 별 파티션 테이블 혹은 대용량 테이블의 split 수집 등의 다양한 방식의 수집이 가능한가?
    * 단일 테이블은 당연히 가능하고, 서버 별 파티션 수집도 반복적인 실행이므로 어려움 없이 포팅이 가능하다
    * 다만 split-by 구현은 sqoop 에서 local 환경에서 수행이 가능한지 테스트가 필요하지만 가능할 것으로 추측한다
  * 최대 수용 가능한 장비의 수와 수평적인 확장이 가능한지, 특정 노드에 몰리는 등의 스케줄링 기능이 가능할까?
    * docker swarm job coordinator 의 기능을 활용하는 기술 검토를 통해 리소스 코디네이팅이 해보면 좋겠다
    * 결국 yarn 은 실행 중인 작업의 **상태 관리**, **스케줄링** 및 **리소스**(CPU, Memory) 관리가 목적이다
    * 이러한 3가지 기능 + **네트워크** 사용 까지 고려한 소프트웨어 실행 관리를 구현하는 것을 고민해보자
    * 일전에 검토 했던 논리 테이블 + 파티션 수 등을 고려하여 실행 가능한 도커 컨테이너의 최대 용량을 고려하여 실행하는 소프트웨어적인 스케줄러 개발
    * 해당 작업을 수행하는 녀석은 앱마스터 역할이고 데몬 프로세스로 수행되고 모든 작업이 완료될 때까지 떠 있으면서 관리를 담당한다
    * 즉, 상태 관리, 실행 중인 작업의 수를 별도의 메타데이터로 저장 관리해야 하고 이를 통해서 작업 스케줄링을 할 수 있다
    * 별도 API 서버는 몽고디비를 저장소로 사용하고 별도 처리를 위한 메타데이터를 관리한다
      * 상태관리는 기본적으로 국내 API 서버 호출을 통해서 관리되어야 하고  
      * 리소스 (CPU, Memory, 네트워크 등)을 고려한 스케줄링은 별도의 메타데이터 및 상태 관리를 활용하여 수행되도록 구현해야 한다
  * 테이블 단위 혹은 파티션 단위로 성공 실패여부를 국내 상태 관리를 통해서 가능할까?
    * 기존과 같이 구현하되 execute-process 수행 시에 원격지에 메타데이터를 전달하고 이를 받는 API 서버 구현
    * 해당 API 내에서 기존 spark-submit 작업을 수행하고 다시 국내 API 통해서 running 상태로 변경해주고
    * 필요에 따라서 pre-running 같은 상태를 추가해야 할 수도 있겠다
    * 정상적으로 테이블 수집이 완료 혹은 실패 결과를 다시 국내 API 통해서 failed or sucess 상태로 변경해주도록 구현할 수 있다
  * 도커 스웜 구성 시에 5~10대 정도로 노드를 구성하는 경우 api 서버는 3대만 구성하고 7대는 컨테이너로만 구성할 수 있나?
    * 수집을 위한 컨테이너는 7대 사용하고, API 서버만 dedicated 된 3대 구성하고 매니저 노드로 drain 설정하고 vip 구성하면 좋겠다
    * 노드 역할을 라벨로 나누고, API는 3대 고정 배치, Sqoop 워커는 별도 제약으로 배치 (api=true, sqoop=true)
  * 네트워크 호출 및 프로토콜에 대한 표준을 정할 필요는 없는가?
    * 내부 서비스 통신: overlay network + Swarm 서비스명 기반 접근
    * 외부 API 진입: host publish + 외부 VIP/LB

#### 아키텍처 및 운영 방향을 요약해보면?

* 대략적인 프로세스

  * ```
    [국내 Control Plane]
      - 빅브라더 데이터베이스 + 테이블 서비스
      - 빅브라더 테이블 상태관리 API Server
            ↓
    [원격 Execution Plane]
      - 상태관리 API Server (별도 몽고 클러스터를 통한 메타 관리)
      - Docker Swarm
      - Sqoop 스케줄러 컨테이너 (현재 실행 가능한 컨테이너 및 수행 가능한 최대 타스크 수 설정, 노드 당 최대 실행 가느아 타스크 통해 스케줄링)
      - Sqoop 실행 컨테이너
            ↓
    [원격 Storage]
      - Local Storage 저장
            ↓
    [국내 반입]
      - 빅브릿지 rclone 서비스
            ↓
    [Hive / 기존 시스템]
      - 빅브라더 하이브 서비스
    ```
