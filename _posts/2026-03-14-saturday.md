  * 알아서 잘 하는 것인가? 생각해보면 그럴싸 해보이나 결국 내가 생각하는 것을 이해해주고 해주기를 바라는 게 아닐까?
  * 나는 그러면 어떤 것들을 어떻게 요청하고 전달할 수 있었나? 나는 요즘 무엇을 생각하고 하려고 하는가?
  * 결국 평소에 혹은 내가 원하는 데이터 플랫폼을 상상하고 그려나가는 것들이 은연중에 나오는 게 아닐까?
  * 그렇게 생각하면 결국에는 꿈돌이 혹은 상상을 잘 하는 사람이 주변에 영향을 미칠 수 있는게 아닐까?

* 오늘 발생한 장애와 조치에 대한 동료 A 와 B 의 행동을 기준으로 고찰
  * 오늘과 같은 장애가 있는 경우 동료 A 가 하는 모습 처럼, 왜 발생했는지를 끝까지 놓치지 않고 밝혀내려고 하는 자세 "파일이 어떻게 생성되는지 파악하고, 1~2시간 지연 대신 1일 지연으로 적재하는 것을 협의한 점" 등이 다시 발생할 수 있는 문제를 애초에 근절할 수 있었다
  * 반면에 동료 B 는 충분히 적극적인 대처를 열심히 해주었으나, 소극적이라고 해야 할지 아니 ... 근시안 적이라고 할 수도 있겠지만 긍정적으로 바라보면 이용자 관점에서 요구 사항을 만족시키려 했다고 볼 수 있으나, CBT 인점 다시 또 문제가 발생할 수 있는 점 (이게 가장 중요) 등을 바라보지 못하고 판단한 것 때문에 아쉽다고 생각하는 것 같다
  * 이러한 관점에서는 기술적인 요소 보다는 전체적인 일의 흐름이나 상황 그리고 가장 큰 목적인 안정적인 서비스를 제공한다는 것을 체득 혹은 잘 이해하고 있고 어떤 판단이 적합한지를 이해하는 것이다.. 그런데 이런 것들을 가르쳐서 나아질 수 있는 부분인지 모르겠다 결국 나는 강사로써 가이드라인을 제공한다는 면에서 이런 것들을 커리큘럼으로 만들어낼 수는 없기 때문이다
  * 결국 일을 잘하는 사람은 기술적인 이해나 수준 보다는 현재 돌아가는 상황을 잘 이해하고 적절한 판단을 하는 것이 회사에는 유용하다
  * 그렇다면 이러한 지식 혹은 능력을 키우려면 평소에 어떤 행동 학습 혹은 사고를 해야 성장할 수 있을까?

  * 로그 혹은 테이블 등의 데이터의 특성에 따른 수집 방식과 제공 가능한 서비스에 대한 판단

  * 전달 혹은 지시한 내용을 기록하지 않거나 기억하지 못하여 누락 지연되는 상황에 대한 문제점
  * 무언가 상황을 전달할 때에 너무 장황하거나 자세한 경우, 몇 번 반복하더라도 핵심을 짚어내지 못하는 것
  * 스트리밍 데이터 수집 vs. 스냅샷 데이터 수집 차이점에 대한 스케줄링 및 해결 방안에 대한 기술 검토

* 뜬금 없지만 테이블 수집 성능 개선에 대한 고찰을 해볼까 한다
  * 개별 테이블의 상태 정보에 해당 테이블의 특징을 표현할 수 있는 정보를 남길 수 있다면 좋겠다 예를 들어 어느 정도의 튜플을 가지고 평균적으로 얼마나 소요되는지, 몇 개의 파티션이 존재하는 지, 국내인지 국외인지 수집의 편차가 얼마나 되는지, 실패 횟수는 얼마나 되는지 등
    * table_name
    * source_system
    * priority
    * sla_deadline
    * estimated_size_bytes
    * estimated_duration_sec
    * table_class (XL/L/M/S/XS)
    * max_parallelism_per_source
    * network_weight
    * last_7d_avg_duration
    * last_7d_avg_bytes
    * last_result_status

  * 무엇보다 하나의 논리적인 테이블이 가질 수 있는 파티션의 수가 100개가 넘어가고 동시에 수행하는 경우 네트워크 버스팅이 당연하게도 일어나게 되는데 ... 큐를 제한 하는 것만으로 네트워크 버스팅을 막을 수 있을지도 의문이고, 그렇다면 슬로우 스타트를 하는 것이 도움이 되는지도 모르겠다
    * 위의 매트릭을 활용하여 테이블의 수준을  default (통계정보 미수집) , light, medium, heavy 와 같이 구별한다
    * 글로벌 제약 예시
      * Heavy 논리 테이블: 동시에 2개
      * Medium 논리 테이블: 동시에 5개
      * Light 논리 테이블: 동시에 10개
    * 로컬 제약 예시
      * Heavy: 활성 서버 파티션 20개
      * Medium: 활성 서버 파티션 40개
      * Light: 활성 서버 파티션 80개
    * 이러한 경험적인 수치를 통해서 해당 값을 최적화하는 문제로 접근하면 좋겠다
      * 결국 테이블 수집에 의해서 발생하는 네트워크 사용 최대치를 수치화 해서 수집할 수 있어야 최적화가 가능하다
      * **네트워크 피크 완화** (시간 단위 네트워크 총 사용량의 p95 / p99, 초단위 burst peak, QoS throttle 발생 횟수, 네트워크 경보 횟수)
      * **큐 안정성** (동시 실행 app 수의 변동성, 활성 파티션 수의 p95, heavy class 동시 실행 수, backlog 누적량)
      * **실패/재시도 감소** (timeout rate, source side reject / connection failure, app retry rate, partial failure rate)
      * 등과 같이 기대하는 네트워크 사용의 그림을 측정할 수 있는 지표가 반드시 측정 가능하고 목표 시스템의 기준을 정해야 한다
    * 가장 심플한 POC 를 위한 기능을 설계하고 구현해볼까?
      * 데이터 크기와 서버 파티션 수를 기준으로 테이블 등급을 메타데이터에 부여
      * 기존에 수행되는 테이블 동시 수를 감안하여 글로벌 전체 실행 가능 논리 테이블 수, 헤비 테이블 한정 로컬 파티션 수 선정
      * 게임 별로는 동시에 수행되지 않으나, 운이 나빠서 대용량 테이블이 한 번에 스케줄링 되는 것을 제어하기는 어렵다
      * 이러한 접근이 가능하려면 스케줄러가 테이블 수준에서 실행 자체도 컨트롤 할 수 있도로 구현되어야 한다
      * 하나의 스케줄에 포함된 전체 작업을 수행 하는 것을 리소스를 고려해서 실행 할 수 있도록 구현할 수 있어야 한다
      * 결국 **스케줄 하나가 최대한 네모진 사각형을 잘 사용할 것을 가정**하고 스케줄 수준에서 겹치지 않음을 가정
      * 스케줄 하나는 임계치를 통해서 주어진 네트워크 대역을 다 사용할 수 있음을 가정하고 네트워크 대역 네모를 다 쓰도록 최적화
        * 어차피 동시에 모든 작업을 제출할 수 있고, 
        * 글로벌 임계치 : 
        * 로컬 임계치 : 
      * 글로벌 정보 메타 테이블 설계, 로컬 제약은 메타를 읽어서 고정된 값으로 수행하도록 구현
